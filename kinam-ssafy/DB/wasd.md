# Bitmap Index 상세 설명

## 카디널리티(Cardinality)란?

### 정의
| 개념 | 설명 |
|------|------|
| **카디널리티** | 컬럼에서 **중복을 제거한 고유한 값의 개수** |
| **높은 카디널리티** | 고유 값이 많음 (주민번호, 이메일 등) |
| **낮은 카디널리티** | 고유 값이 적음 (성별, 지역, Y/N 등) |

### 카디널리티 비교
| 컬럼 | 전체 데이터 | 고유 값 개수 | 카디널리티 |
|------|----------|------------|----------|
| **주민번호** | 1,000,000명 | 1,000,000개 | **매우 높음** |
| **이메일** | 1,000,000명 | 950,000개 | **높음** |
| **나이** | 1,000,000명 | 100개 | 중간 |
| **지역** | 1,000,000명 | 17개 | **낮음** |
| **성별** | 1,000,000명 | 2개 | **매우 낮음** |

## Bitmap Index 원리

### 기본 구조

#### 예시: 100만 명의 성별 데이터
```sql
users 테이블 (1,000,000 rows)
| id  | name | gender |
|-----|------|--------|
| 1   | 홍길동 | M     |
| 2   | 김영희 | F     |
| 3   | 박철수 | M     |
| ... | ...  | ...   |
```

#### Bitmap Index 생성
```
gender 컬럼의 Bitmap Index:

M (남성):  [1, 0, 1, 0, 1, 1, 0, 1, ...]
           ↑  ↑  ↑  ↑  ↑  ↑  ↑  ↑
          id1 id2 id3 id4 id5 id6 id7 id8

F (여성):  [0, 1, 0, 1, 0, 0, 1, 0, ...]
           ↑  ↑  ↑  ↑  ↑  ↑  ↑  ↑
          id1 id2 id3 id4 id5 id6 id7 id8

1 = 해당 값, 0 = 다른 값
```

## Bitmap Index의 장점

### 1. 압축률이 매우 높음

#### B-Tree Index (일반 인덱스)
```
성별 인덱스 저장:
M → [1, 3, 5, 6, 8, 10, 11, 13, ...]  ← ID 리스트
F → [2, 4, 7, 9, 12, 14, 15, ...]     ← ID 리스트

100만 건 × 4byte(ID) = 4MB
```

#### Bitmap Index
```
M: [101101011010...]  ← 비트로 표현
F: [010010100101...]  ← 비트로 표현

100만 건 × 1bit = 125KB
압축률: 약 32배 절약!
```

### 저장 공간 비교
| 인덱스 종류 | 100만 건 저장 용량 | 압축률 |
|-----------|-----------------|--------|
| B-Tree | 약 4MB | 1배 |
| **Bitmap** | **약 125KB** | **32배 절약** |

---

### 2. 다중 조건 검색에 매우 빠름

#### 쿼리 예시
```sql
SELECT * FROM users 
WHERE gender = 'M' 
  AND region = '서울'
  AND status = 'active';
```

#### Bitmap 연산 (비트 AND 연산)
```
gender='M':     [1,0,1,0,1,1,0,1,0,1]
region='서울':  [1,1,0,0,1,0,0,1,1,0]
status='active':[1,0,1,1,1,0,1,1,0,1]
───────────────────────────────────── AND 연산
결과:           [1,0,0,0,1,0,0,1,0,0]
                ↑        ↑     ↑
               id1      id5   id8 찾음!
```

#### 연산 속도
| 작업 | 일반 Index | Bitmap Index |
|------|-----------|-------------|
| 각 조건 검색 | 3번의 Index Scan | 3개의 비트맵 로드 |
| 결과 결합 | ID 리스트 비교 (느림) | **비트 AND 연산 (매우 빠름)** |
| 복잡도 | O(n) | **O(1) - CPU 명령어 1번** |

---

### 3. 집계 연산이 빠름

```sql
SELECT COUNT(*) FROM users WHERE gender = 'M';
```

#### B-Tree Index
```
M에 해당하는 ID 리스트를 모두 세어야 함
→ ID 개수만큼 카운트 연산
```

#### Bitmap Index
```
M 비트맵: [1,0,1,0,1,1,0,1,...]
→ 1의 개수만 세면 됨 (비트 카운트 연산 - 매우 빠름)
```

## Bitmap Index 적합한 경우

### ✅ 적합한 경우
| 조건 | 이유 |
|------|------|
| **낮은 카디널리티** | 고유 값이 적을수록 압축률↑ |
| **읽기 작업 많음** | 빠른 검색 및 집계 |
| **다중 조건 검색** | 비트 연산으로 빠른 결합 |
| **대용량 데이터** | 압축으로 저장 공간 절약 |

### ❌ 부적합한 경우
| 조건 | 이유 |
|------|------|
| **높은 카디널리티** | 비트맵이 너무 많아짐 |
| **쓰기 작업 많음** | UPDATE 시 비트맵 전체 재구성 필요 |
| **OLTP 환경** | 빈번한 변경 불리 |

# 논클러스터드 Index의 포인터

## 포인터가 있는 이유

### 클러스터드 Index (포인터 없음)
```
인덱스 = 실제 데이터 위치

[B-Tree 구조]
    [50]
   /    \
[25]    [75]
 ↓       ↓
실제 데이터 저장 위치
(디스크의 물리적 위치)
```

### 논클러스터드 Index (포인터 있음)
```
인덱스 → 포인터 → 실제 데이터

[B-Tree 구조]         [포인터]      [실제 데이터]
    [name:B]     →    0x1A2B  →   [id:1, name:B, age:25]
       /\
[name:A][name:C] →    0x3C4D  →   [id:3, name:C, age:30]
```

---

## 포인터의 장점

### 1. 여러 개 생성 가능

#### 클러스터드 (테이블당 1개만)
```
users 테이블의 물리적 정렬:
id 순서로만 정렬됨 (변경 불가)

[1, 홍길동, 25]
[2, 김영희, 30]
[3, 박철수, 22]
↑ id 기준으로 물리적 정렬
```

#### 논클러스터드 (여러 개 가능)
```
users 테이블 (id 순서 유지):
[1, 홍길동, 25]  ← 0x1000 주소
[2, 김영희, 30]  ← 0x2000 주소
[3, 박철수, 22]  ← 0x3000 주소

name 인덱스 (별도):          age 인덱스 (별도):
[김영희] → 0x2000           [22] → 0x3000
[박철수] → 0x3000           [25] → 0x1000
[홍길동] → 0x1000           [30] → 0x2000

→ 원본 테이블은 그대로, 인덱스만 각자 정렬!
```

| 특징 | 클러스터드 | 논클러스터드 |
|------|-----------|-------------|
| 개수 제한 | 1개만 | **여러 개** |
| 테이블 변경 | 물리적 재정렬 | **변경 없음** |
| 생성 영향 | 큼 | 작음 |

---

### 2. 테이블 구조 독립성

#### 클러스터드 Index
```
name으로 검색하고 싶다?
→ 전체 테이블을 name 순서로 물리적 재배치 필요!
→ 기존 id 순서는 망가짐
→ 불가능!
```

#### 논클러스터드 Index
```
name으로 검색하고 싶다?
→ name 인덱스만 별도로 생성
→ 원본 테이블은 그대로 유지
→ 가능! 몇 개든 추가 가능
```

---

### 3. 유연한 검색 경로

```sql
-- 시나리오 1: id로 검색
SELECT * FROM users WHERE id = 100;
→ 클러스터드 Index 사용 (직접 접근)

-- 시나리오 2: name으로 검색
SELECT * FROM users WHERE name = '홍길동';
→ 논클러스터드 Index 사용 (포인터 경유)

-- 시나리오 3: age로 검색
SELECT * FROM users WHERE age = 25;
→ 또 다른 논클러스터드 Index 사용
```

| 검색 조건 | 사용 Index | 경로 |
|----------|-----------|------|
| id | 클러스터드 | 직접 |
| name | 논클러스터드 | 인덱스 → 포인터 → 데이터 |
| age | 논클러스터드 | 인덱스 → 포인터 → 데이터 |

---

## 포인터의 단점

### 추가 조회 단계
```
클러스터드 (1단계):
Index 탐색 → 데이터 바로 읽음

논클러스터드 (2단계):
Index 탐색 → 포인터 확인 → 포인터 위치로 이동 → 데이터 읽음
                      ↑ 추가 단계!
```

### 성능 비교
| Index 종류 | 검색 단계 | 속도 |
|-----------|----------|------|
| 클러스터드 | 1단계 | 빠름 ★★★★★ |
| 논클러스터드 | 2단계 | 보통 ★★★ |

---

## 실생활 비유

### 클러스터드 = 사전
```
단어가 가나다순으로 정렬됨
"사과" 찾기 → 'ㅅ' 섹션 가면 바로 있음
장점: 빠름
단점: 한 가지 순서만 가능 (가나다순만)
```

### 논클러스터드 = 교과서 + 색인
```
본문: 페이지 순서대로 (1, 2, 3...)
색인: "광합성 → 45페이지"
      "뉴턴 → 102페이지"

장점: 여러 색인 가능 (인물 색인, 용어 색인...)
단점: 색인 → 페이지 이동 (2단계)
```

## 핵심 정리

### Bitmap Index
| 항목 | 내용 |
|------|------|
| **카디널리티** | 고유 값 개수 (낮을수록 효율↑) |
| **원리** | 각 값마다 비트맵 생성 (1=해당, 0=미해당) |
| **장점** | 압축률 32배, 다중 조건 빠름, 집계 빠름 |
| **적합** | 성별, 지역, Y/N 등 + 읽기 위주 |

### 논클러스터드 포인터
| 항목 | 내용 |
|------|------|
| **포인터 역할** | 인덱스 → 실제 데이터 위치 연결 |
| **장점 1** | 여러 개 생성 가능 |
| **장점 2** | 테이블 물리 구조 변경 없음 |
| **단점** | 검색 시 2단계 (인덱스→포인터→데이터) |
